<objective>
Build the user confirmation flow module that gates all SQLite writes behind explicit user approval.

Purpose: Agent findings are probabilistic. The confirmation flow is the trust boundary — nothing enters the graph until a human has reviewed and approved it. Grouping by confidence prevents rubber-stamping (Pitfall 9 from PITFALLS.md: 50+ individual prompts cause users to approve everything without reading).

Output:
- `worker/confirmation-flow.js` — pure module with all confirmation logic (no I/O, no SQLite calls)
- `worker/confirmation-flow.test.js` — tests covering grouping, the 10-finding cap on low confidence, edit application, and prompt formatting
</objective>

Agent findings structure (produced by Phase 18 scan-manager):
```javascript
// Single finding object
{
  service: "user-service",
  repo: "/path/to/repo",
  connections: [
    {
      sourceService: "auth-service",
      targetService: "user-service",
      protocol: "rest",         // rest | grpc | kafka | rabbitmq | internal | sdk
      method: "GET",
      path: "/users/:id",
      sourceFile: "auth/client.ts:fetchUser",
      targetFile: "users/routes.ts:getUser"
    }
  ],
  schemas: [
    {
      connectionId: null,       // null at pre-persist stage; db assigns after write
      role: "response",         // request | response | event_payload
      name: "UserResponse",
      file: "users/types.ts",
      fields: [
        { name: "id", type: "string", required: true },
        { name: "email", type: "string", required: true }
      ]
    }
  ],
  confidence: "high" | "low"   // from agent (Phase 18 SCAN-04)
}
```

Confirmation flow design (from PITFALLS.md Pitfall 9 and UCON requirements):
- HIGH confidence: batch confirm — show summary grouped by repo, one confirm covers all
- LOW confidence: show individually with specific clarifying question; cap at MAX_LOW_CONFIDENCE=10
- User can edit: findings are presented as a JSON-like text representation the user can modify
- Gate: Phase 20 command layer must only call db.writeScan(findings) AFTER this module returns confirmed findings

Edit format the user sees (to enable free-form editing before confirm):
```
--- High confidence findings (N connections across M repos) ---
[repo: ../api]
  user-service EXPOSES GET /users/:id (rest) → UserResponse schema
  auth-service CONSUMES GET /users/:id (rest)

Type "confirm" to accept all high-confidence findings, or describe changes.

--- Low confidence finding 1 of N ---
  Possible connection: payment-service → order-service via POST /orders
  Evidence: string "/orders" found in payment/client.ts:53
  Question: Is payment-service intentionally calling order-service's POST /orders endpoint?
  Type "yes", "no", or describe the correct connection.
```
