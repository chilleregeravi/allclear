  <name>Task 1: Agent prompt template</name>
  <files>worker/agent-prompt.md</files>
  <behavior>
    - Prompt produces JSON with top-level keys: service_name, confidence, services[], connections[], schemas[]
    - Each connection has: source, target, protocol (rest|grpc|kafka|rabbitmq|internal|sdk), method, path, source_file, target_file, confidence (high|low), evidence
    - Each schema has: name, role (request|response|event_payload), file, fields[]
    - Each field has: name, type, required (boolean)
    - confidence=high only when a literal string endpoint path is present in source; confidence=low for inferred or dynamic paths
    - Prompt instructs agent NOT to infer from usage patterns, only report literal definitions
    - Prompt explicitly instructs agent to report template paths (e.g., /users/{id}) not interpolated instances
  </behavior>
  <action>
Create `worker/agent-prompt.md` — the Markdown template injected into every agent scan invocation.

The prompt must:
1. Instruct the agent to scan ALL files in the repo path (no file type restriction — language-agnostic per SCAN-08).
2. Tell the agent to identify service boundaries by looking for: HTTP server setup, route definitions, gRPC service declarations, event topic producer/consumer registrations, internal SDK exports.
3. Require the agent to return ONLY a JSON object matching the findings schema (no prose explanation before or after the JSON block). Agent output must be a fenced ```json block followed by nothing else.
4. Define confidence rules clearly:
   - HIGH: agent found a literal string definition of the endpoint path or event topic in source code (e.g., `@app.route('/users')`, `router.get('/health', ...)`)
   - LOW: agent inferred the connection from a variable, dynamic construction, or usage pattern without a literal definition
5. Include per-connection `evidence` field: the exact code snippet (≤ 3 lines) that led to the finding. This is the secondary validation hook.
6. Explicitly prohibit the agent from inventing connections. Quote: "If you cannot find a literal string definition, do not report the endpoint. Report only what you can directly cite from source."
7. Include the full JSON output schema inline in the prompt (copy from findings-schema.js structure) so the agent knows exactly what to produce.
8. The prompt template uses `{{REPO_PATH}}` and `{{SERVICE_HINT}}` as interpolation tokens that the scan manager replaces at runtime.

Example output the prompt should elicit:
```json
{
  "service_name": "user-api",
  "confidence": "high",
  "services": [
    {
      "name": "user-api",
      "root_path": "src/",
      "language": "typescript",
      "confidence": "high"
    }
  ],
  "connections": [
    {
      "source": "user-api",
      "target": "auth-service",
      "protocol": "rest",
      "method": "POST",
      "path": "/auth/validate",
      "source_file": "src/middleware/auth.ts:validateToken",
      "target_file": null,
      "confidence": "high",
      "evidence": "const res = await fetch('/auth/validate', { method: 'POST' })"
    }
  ],
  "schemas": [
    {
      "name": "UserCreateRequest",
      "role": "request",
      "file": "src/types/user.ts",
      "fields": [
        { "name": "email", "type": "string", "required": true },
        { "name": "name", "type": "string", "required": true },
        { "name": "role", "type": "string", "required": false }
      ]
    }
  ]
}
```

<task type="auto" tdd="true">
  <name>Task 2: Findings schema and validator</name>
  <files>worker/findings-schema.js, worker/findings-schema.test.js</files>
  <behavior>
    - validateFindings(obj) returns { valid: true, findings: obj } for well-formed input
    - validateFindings(null) returns { valid: false, error: "findings must be an object" }
    - validateFindings({}) returns { valid: false, error: "missing required field: connections" }
    - validateFindings with unknown protocol returns { valid: false, error: "connection[0].protocol must be one of: rest, grpc, kafka, rabbitmq, internal, sdk" }
    - validateFindings with confidence not in (high, low) returns { valid: false, error: ... }
    - validateFindings with valid minimal input (no schemas, no fields) returns valid: true
    - parseAgentOutput(rawText) extracts the fenced ```json block from agent prose and calls validateFindings
    - parseAgentOutput with no JSON block returns { valid: false, error: "no JSON block found in agent output" }
  </behavior>
  <action>
Create `worker/findings-schema.js` with:

1. `VALID_PROTOCOLS` constant: `['rest', 'grpc', 'kafka', 'rabbitmq', 'internal', 'sdk']`
2. `VALID_CONFIDENCE` constant: `['high', 'low']`
3. `VALID_ROLES` constant: `['request', 'response', 'event_payload']`
4. `validateFindings(obj)` — plain-object validator (no external library, zero dependencies):
   - Validates required top-level keys: service_name (string), confidence (high|low), services (array), connections (array), schemas (array)
   - For each connection: validates source (string), target (string), protocol (VALID_PROTOCOLS), method (string), path (string), source_file (string|null), confidence (VALID_CONFIDENCE), evidence (string). target_file is optional.
   - For each schema: validates name (string), role (VALID_ROLES), file (string), fields (array)
   - For each field: validates name (string), type (string), required (boolean)
   - Returns `{ valid: true, findings: obj }` or `{ valid: false, error: string }`
5. `parseAgentOutput(rawText)` — extracts the first fenced ```json block from raw agent output string and calls validateFindings on the parsed object. Returns same shape as validateFindings.
   - Handles leading/trailing prose (agents sometimes explain before the JSON block)
   - Returns { valid: false, error: "no JSON block found in agent output" } if no fenced block found
   - Returns { valid: false, error: "JSON parse error: ..." } if JSON.parse throws

Use JSDoc `@typedef` annotations for the FindingsResult, Connection, Schema, Field types so downstream files get IDE hints without TypeScript compilation.

Create `worker/findings-schema.test.js` using Node.js built-in `node:test` (no external test runner — keeps zero-dep constraint). Tests must cover all behavior cases above.
  </action>
  <verify>
    <automated>node --test worker/findings-schema.test.js</automated>
  </verify>
  <done>All test cases pass; validateFindings and parseAgentOutput exported correctly; zero external dependencies (no require beyond node builtins)</done>
</task>
