# Plan 118-02: `/arcanon:rescan` — Single-Repo Re-Scan

**Phase:** 118 · **Plan:** 02 · **Owned REQs:** CORRECT-04, CORRECT-05, CORRECT-07 (rescan bats happy-path + error case) · **Risk:** Medium · **Wave:** 3

## Goal

`/arcanon:rescan <repo-path-or-name>` re-scans exactly one repo, bypasses the incremental change-detection skip path, updates `scan_versions` for that repo only, and (if Phase 117's `applyPendingOverrides` hook is in place) consumes any pending overrides for the repo's findings during the re-scan. Other repos in the project's linked-repos config are not touched.

## Success Criteria (what must be TRUE after this plan ships)

1. `/arcanon:rescan ../api` resolves the path relative to `cwd`, looks up the matching `repos.id`, and triggers a full re-scan of that one repo only. Exit 0 on success.
2. `/arcanon:rescan api` (a name match against `repos.name`) resolves to the same repo as `/arcanon:rescan ../api` when both refer to the same registered repo.
3. `/arcanon:rescan nonexistent-repo` exits 2 with a friendly error listing every available repo in the project (`repos.name` + `repos.path`).
4. Multi-match by name (two repos in the project share the same `basename`) exits 2 with a disambiguation message listing each match by `(name, path)` and instructing the user to use the full path.
5. The re-scan ALWAYS runs in `full` mode regardless of whether the repo's HEAD commit equals `last_scanned_commit` — i.e., bypasses the `mode: 'skip'` short-circuit at `manager.js:654-657`.
6. After the rescan, `scan_versions` has a new row for the rescanned repo with a fresh `id` and current `completed_at`. Other repos' `scan_versions` rows are untouched (no new entries for them).
7. If the worker is not running, `/arcanon:rescan` exits 1 with the same friendly "Worker not running" message that `/arcanon:verify` uses (`hub.js:445-453`). Does NOT auto-start the worker.
8. `/arcanon:rescan --json ...` emits `{ok: true, repo_id, repo_path, repo_name, scan_version_id, mode: "full"}` instead of the human line.
9. `/arcanon:rescan` is silent in non-Arcanon directories (no `impact-map.db`) and exits 0 — same contract as `/arcanon:list`.
10. `commands/rescan.md` exists with frontmatter (`description`, `argument-hint`, `allowed-tools: Bash`) and a body that detects worker, sources `lib/worker-client.sh`, calls `bash hub.sh rescan $ARGUMENTS`.
11. Bats test asserts: happy path (path arg), happy path (name arg), nonexistent repo exits 2.

## <assumptions_about_phase_117>

This plan does NOT consume any Phase 117 deliverable directly. The rescan path runs `scanRepos([repoPath], {full: true}, qe)` end-to-end. If Phase 117's `applyPendingOverrides` hook is wired into `scanRepos` (between `persistFindings` and `endScan` per the ROADMAP PRE-FLIGHT for Phase 117), then any pending `scan_overrides` rows for the rescanned repo will be applied as a side-effect of the re-scan — which is the intended end-to-end behavior (the user runs `/arcanon:correct ...` then `/arcanon:rescan ...` and expects the corrections to take effect).

| # | Assumption | Where it's used in this plan |
|---|-----------|-----|
| **P117-A'** | `scanRepos([repoPath], {full: true}, qe)` runs to completion against a DB that has migration 017 applied (table exists). The `applyPendingOverrides` hook reads `scan_overrides WHERE applied_in_scan_version_id IS NULL` and stamps applied rows. | Task 1 — the new `scanSingleRepo` wrapper just delegates to `scanRepos`; if 117's apply hook is present in `scanRepos`, rescan inherits the apply behavior. |
| **P117-B'** | The `applyPendingOverrides` hook is idempotent — re-running `/arcanon:rescan` with no new overrides produces the same scan output as the first rescan (no double-apply, no constraint violation). | Task 5 (bats happy path can run rescan twice if desired without breaking; not strictly tested in this plan). |
| **P117-C'** | If Phase 117 has NOT shipped at execution time (table missing), `scanRepos` works exactly as it does in v0.1.3 — rescan still runs successfully, just without applying any overrides. The migration-017-aware code path in 117 must degrade gracefully when the table is absent (or the migration must be present in the DB). | Task 5 (bats test does not assume overrides are applied; it only asserts that a new `scan_versions` row is created for the rescanned repo). |

**Plan 118-02 is therefore loosely coupled to Phase 117** — it ships a single-repo scan trigger, which is a complete unit of value on its own (the user gets "rescan one repo without touching others"). When 117 ships, the override-apply happens for free during the rescan.

</assumptions_about_phase_117>

## Tasks (executable, ordered)

### Task 1 — Add `scanSingleRepo` to `worker/scan/manager.js`

**File:** `plugins/arcanon/worker/scan/manager.js` (modify — add export)

Add a new exported function near the bottom of the file (after `scanRepos`, before the `_readHubAutoSync` re-export at line ~961):

```javascript
/**
 * scanSingleRepo — re-scan exactly one repo, bypassing the incremental skip.
 *
 * Used by /arcanon:rescan (CORRECT-04). Always invokes scanRepos with
 * options.full = true, which short-circuits buildScanContext's "mode: skip"
 * branch (manager.js:393-396) and forces the full scan path. Other repos
 * registered in the project's linked-repos config are NOT scanned.
 *
 * Phase 117 note: when applyPendingOverrides is wired into scanRepos
 * (between persistFindings and endScan), rescanning will also apply any
 * pending scan_overrides rows for this repo as a side-effect — which is
 * the intended /arcanon:correct → /arcanon:rescan workflow.
 *
 * @param {string} repoPath - Absolute path to the repo (must already be
 *   registered in the repos table; caller resolves identifier → path).
 * @param {{ upsertRepo, getRepoState, beginScan, persistFindings, endScan, _db }} queryEngine
 * @param {{ }} [options] - Forwarded to scanRepos (full is force-set true).
 * @returns {Promise<import('./manager.js').ScanResult>}
 */
export async function scanSingleRepo(repoPath, queryEngine, options = {}) {
  // CORRECT-04: bypass incremental skip — always full-scan a single repo.
  const results = await scanRepos([repoPath], { ...options, full: true }, queryEngine);
  return results[0]; // single ScanResult — caller doesn't need the array shape
}
```

**Done when:** `import { scanSingleRepo } from "./manager.js"` works; calling it with a real repo path + queryEngine triggers a full scan of that repo only (verifiable by checking that only one row appears in `scan_versions` post-scan when the project has multiple repos and only one was rescanned).

### Task 2 — Add `POST /api/rescan` endpoint to `worker/server/http.js`

**File:** `plugins/arcanon/worker/server/http.js` (modify)

Add a new route handler after the existing `POST /scan` handler (~line 525). The endpoint:

- **Method:** `POST`
- **Path:** `/api/rescan`
- **Query params:** `project=<absolute-root>` (resolves the project DB), `repo=<identifier>` (the repo path or name to rescan)
- **Response (success):** `200 {ok: true, repo_id, repo_path, repo_name, scan_version_id, mode: "full"}`
- **Response (repo not found):** `404 {error: "repo '<identifier>' not found", available: [{name, path}, ...]}`
- **Response (multi-match by name):** `409 {error: "name '<identifier>' matches multiple repos", matches: [{id, path}, ...]}`
- **Response (worker bootstrap not ready, e.g., `agentRunner` not injected):** `503 {error: "worker not ready: agentRunner not initialized"}`

Handler responsibilities, in order:

1. Resolve `project` query param via `getQE(request)` (existing pattern, see `http.js:161-167`).
2. Read `repo` query param (URL-decode if needed).
3. Call a new helper `_resolveRepoIdentifier(repoArg, qe, projectRoot)` (defined inline or in `worker/lib/repo-resolver.js`):
   - Try `path.resolve(projectRoot, repoArg)` then `SELECT id, path, name FROM repos WHERE path = ?`.
   - If no match, try `SELECT id, path, name FROM repos WHERE name = ?` (returns array). Zero results → 404. >1 results → 409.
   - One result → return `{id, path, name}`.
4. Call `scanSingleRepo(repo.path, qe, {})` from `worker/scan/manager.js`.
5. Extract `scan_version_id` from the result (the `scanRepos` result object has `findings` set on success — for `scan_version_id`, query `SELECT id FROM scan_versions WHERE repo_id = ? ORDER BY id DESC LIMIT 1` after the scan completes; or extend the ScanResult shape to expose `scanVersionId`).
6. Reply with the success JSON.

**Important:** the `agentRunner` injection — `scanRepos` throws if `agentRunner === null` at `manager.js:604-606`. Verify in the handler or rely on the worker's normal startup path having injected it (the worker's `index.js` injects the runner before binding to the port). If the worker is up, agentRunner is wired.

**Done when:** `curl -X POST "http://127.0.0.1:37888/api/rescan?project=$(pwd -P)&repo=api"` returns `{ok: true, ...}` against a project with a registered `api` repo.

### Task 3 — Add `cmdRescan` to `worker/cli/hub.js`

**File:** `plugins/arcanon/worker/cli/hub.js` (modify)

Add `async function cmdRescan(flags, positional)` near `cmdRescan` and `cmdCorrect`. Function body:

1. **Project detection** (silent no-op):
   ```javascript
   const cwd = process.cwd();
   const dbPath = path.join(projectHashDir(cwd), "impact-map.db");
   if (!fs.existsSync(dbPath)) process.exit(0);
   ```

2. **Read positional arg.** `const repoArg = positional[0];` — if missing/empty, print usage and exit 2 (`error: /arcanon:rescan requires a repo path or name as the first argument`).

3. **Resolve worker port** (mirror `cmdVerify` at `hub.js:427-438`).

4. **Worker-down detection.** Try a quick `GET /api/readiness`. If unreachable, print the friendly message verbatim from `cmdVerify` (`hub.js:445-453`) and exit 1. Do NOT auto-start the worker (matches the verify contract — rescan is a foreground operator action; worker lifecycle is owned by `/arcanon:map`).

5. **POST to `/api/rescan`:**
   ```javascript
   const params = new URLSearchParams();
   params.set("project", cwd);
   params.set("repo", repoArg);
   const url = `http://127.0.0.1:${workerPort}/api/rescan?${params.toString()}`;
   const r = await fetch(url, { method: "POST" });
   ```

6. **Translate HTTP status to exit code:**
   - 200 → emit success JSON or human line, exit 0
   - 404 (not found) → emit error from response body, exit 2
   - 409 (multi-match) → emit disambiguation from response body, exit 2
   - 503 (worker not ready) → emit "worker bootstrap incomplete", exit 1
   - Other 4xx/5xx → emit response body's `error` field, exit 1
   - Network error → emit "worker not running" message, exit 1

7. **Human output on success:**
   ```
   Rescanned: api (repo_id=3, scan_version_id=42)
   Mode: full (incremental skip bypassed)
   ```

8. **JSON output:** the response body verbatim (already in the right shape per Task 2).

**Register in HANDLERS:**

```javascript
const HANDLERS = {
  // ... existing ...
  correct: cmdCorrect, // CORRECT-02 (Plan 118-01)
  rescan: cmdRescan,   // CORRECT-04 (Plan 118-02)
};
```

**Done when:** `node plugins/arcanon/worker/cli/hub.js rescan api` against a running worker triggers a real re-scan and prints the success line; `node ... rescan nonexistent` exits 2 with the friendly listing.

### Task 6 — Bats E2E tests for `/arcanon:rescan`

**File:** `tests/rescan.bats` (new at repo root, per hard constraint)

Clone the verify.bats / list.bats helper layout. Use port 37996 (avoid collisions: verify=37999, list=37998, correct=37997).

**Fixture:** `plugins/arcanon/tests/fixtures/rescan/seed.sh` + `seed.js` (new). The seed must:
1. Create two real git repos under `$PROJECT_ROOT` (e.g., `repo-a/` and `repo-b/`), each with a single committed file. (Real git is required because `getCurrentHead()` at `manager.js:374-378` shells out to `git -C <path> rev-parse HEAD`.)
2. Open the project DB via `openDb(projectRoot)`.
3. Insert two repo rows (`upsertRepo`) — one for each. Capture both IDs.
4. Insert a `scan_versions` row + `repo_state` row for each repo with `last_scanned_commit = <currentHEAD>` so that without the rescan bypass, the scan would be skipped.
5. Print both `(repo_id, repo_path, repo_name)` tuples on stdout for the test to capture.

**Mock agentRunner:** the worker's startup must use a mock agent runner that returns valid scan JSON quickly (to avoid actually spawning Claude). Look at how `tests/verify.bats` handles this — verify.bats does NOT trigger a scan, so it doesn't need a mock. Rescan DOES trigger a scan, so we need to inject a mock.

**Mock injection mechanism:** the worker's `index.js` injects the agent runner. For the bats test, set an env var like `ARCANON_TEST_AGENT_RUNNER=1` that the worker's bootstrap reads to install a stub runner that returns a hard-coded valid scan response. Add a small one-line code path to the worker's startup. (Alternative: spawn a worker variant from a test-only entrypoint script that wires the stub. The env-var approach is simpler.)

If the env-var injection is too invasive for this plan, a fallback is to write the test to hit `/api/rescan` directly via curl (skipping the CLI wrapper) and let the test inject the runner programmatically before binding to the port. **Recommend the env-var approach as the canonical mechanism — it makes the same trick available to all future scan-triggering tests.**

**Tests (4 cases):**

- **6a. Happy path by path:** `cd $PROJECT_ROOT && bash hub.sh rescan repo-a` → exit 0; output contains `Rescanned: repo-a`. - **6b. Happy path by name:** `bash hub.sh rescan repo-a` (when the cwd doesn't match `repo-a`'s parent) — should still resolve via name lookup. Same assertions as 6a.

- **6c. Nonexistent repo:** `bash hub.sh rescan totally-not-a-repo` → exit 2; stderr contains `not found`; stderr lists `repo-a` and `repo-b` (the available repos).

- **6d. Worker down:** stop the worker before running `bash hub.sh rescan repo-a` → exit 1; stderr contains `worker not running` (matches verify message).

- **6e. Silent no-op:** `cd $BATS_TEST_TMPDIR/empty-dir && bash hub.sh rescan repo-a` → exit 0, empty stdout (silent contract).

**Done when:** `bats tests/rescan.bats` passes all 5 cases. Each case spawns a fresh worker on port 37996 and seeds a fresh project with two real git repos.

## Out of Scope (explicit)

- Multi-repo rescan (`/arcanon:rescan repo-a repo-b`) — single-repo only per CORRECT-04. Future work.
- A `/arcanon:rescan --since <timestamp>` flag — incremental rescan is explicitly bypassed by this command's contract.
- Auto-starting the worker — clone of verify's "worker not running" path; worker lifecycle stays owned by `/arcanon:map`.
- Override-apply semantics — Phase 117's responsibility. This plan triggers a normal scan; if 117's apply hook is wired, override-apply happens transparently.
- A `--dry-run` flag showing what would be re-scanned without invoking the agent — deferred.
