## Overview

v0.1.4 transforms Arcanon from a scan engine into an operator-facing tool. Nine new slash commands, a universal `--help` surface, scan corrections backed by an idempotent `scan_overrides` table, a validate-before-commit shadow-DB workflow, hub envelope v1.2 with explicit-spec drift, and a curated externals catalog with user extension. 41/41 REQs across 9 phases. 117 commits, +33,305/-371 LOC, ~2 days from milestone start to tag.

## Milestone Summary

**Key Decisions:**

- **Apply-hook seam, not flag** — `applyPendingOverrides(scanVersionId, queryEngine, slog)` is a function-typed injection point in `scanRepos`, not a boolean flag wedged into `persistFindings`/`endScan`. Future override-apply variants (Phase 117 user extension, etc.) compose at the seam without touching either neighbor function.
- **Always-fresh shadow QE (Option B)** — `getShadowQueryEngine` bypasses the pool entirely. Considered Option A (pool-key suffix) — rejected because the cached suffix would still share the underlying `_db` singleton via `openDb()`. Option B sidesteps the singleton problem permanently.
- **WAL sidecars renamed atomically with main DB** — both `-wal` and `-shm` are renamed in the same step as the main file in BOTH backup and promote. SQLite's open-time recovery would otherwise see a stale log against the new main file.
- **`_resetDbSingleton()` underscore-prefixed export** — first explicit escape hatch from `openDb()`'s process-cached singleton. Only `evictLiveQueryEngine` calls it. Underscore marks it internal so future callers think twice.
- **Markdown-orchestrated rescan/shadow-scan (post-execution correction)** — original worker-HTTP shape returned 503 in production because the worker has no agent runtime (Claude agents only run on the host). Resolved by deletion + re-architecture, not by deferral. Zero deferred items at ship.
- **Hub payload v1.2 byte-identity for v1.1 users** — Test M11 asserts `JSON.stringify` byte-equivalence for the `"full"` evidence path. Zero drift for every existing caller.
- **Catalog loader accepts plan-shape + actual-shape** — Phase 121 plan assumed `entries:`; Phase 120 shipped `externals:`. Loader's `normalizeCatalog` accepts both. Single point of adaptation; no 120 file rewrite.
- **Universal help via `lib/help.sh` sourcing** — every command's body sources the extractor inline. Substituted for plan's `bash hub.sh <cmd> --help` (which only handles a subset of commands). HELP-04 bats validates 17/17.

**Architectural Correction (post-execution):**

The original 118-02 / 119-01 implementations drove rescan and shadow-scan through worker HTTP routes (`POST /api/rescan`, `POST /scan-shadow`) that depended on an agent-runner injection inside the worker process. Production worker startup never wires an agent runner — Claude agents only run on the host via Claude Code — so both routes returned `503 worker bootstrap incomplete` in production (originally tracked as deferred item `RESCAN-PROD-01`). Rather than ship that gap, both `commands/rescan.md` and `commands/shadow-scan.md` were re-shaped at commit `222144e` to clone `commands/map.md`'s pattern: the markdown body invokes `Agent` directly for discovery + deep scan, then persists via `openDb` (rescan) or `getShadowQueryEngine` (shadow-scan) inline, calling `applyPendingOverrides` explicitly between `persistFindings` and `endScan`. **Removed:** the two HTTP routes, `cmdRescan` / `cmdShadowScan` CLI handlers, `scanSingleRepo` wrapper, and the `ARCANON_TEST_AGENT_RUNNER=1` test stub. **Kept:** `worker/lib/repo-resolver.js` (still called inline from rescan.md), `getShadowQueryEngine` (still used by shadow-scan persistence), Phase 117's `applyPendingOverrides` hook (still fires between `persistFindings` and `endScan`, now invoked explicitly by the markdown command). The shipped behaviour is functionally equivalent to the original plan; only the dispatch path differs. **Result:** zero production routes return 503; zero deferred items at v0.1.4 ship.
