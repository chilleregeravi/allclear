<objective>
Ship the **pure diff engine** for Phase 115 — two modules under `plugins/arcanon/worker/diff/` that (a) resolve a scan-version selector to a `scan_versions.id` and (b) compute a set-diff of services and connections between two scan IDs across two open DB handles. No CLI surface, no slash command, no markdown — that's Plan 115-02. This plan delivers the reusable engine that Phase 119 (`/arcanon:diff --shadow`) imports verbatim.

Purpose: NAV-04 (the engine half). Operators get the diff capability via Plan 115-02; Phase 119 gets the shadow-DB diff via the same engine with one side's DB handle swapped.

Output: New `worker/diff/scan-version-diff.js` (engine), new `worker/diff/resolve-scan.js` (selector resolver), and exhaustive node tests for both. Read-only: no DB writes, no migrations, no auth surface.

Engine signature is the one promise this plan ships:

```javascript
diffScanVersions(dbA: Database, dbB: Database, scanIdA: number, scanIdB: number)
  -> {services: {added, removed, modified}, connections: {added, removed, modified}, summary, same_scan}

resolveScanSelector(db: Database, selector: string, projectRoot?: string)
  -> {scanId: number, resolvedFrom: string}
```

DB handles are passed in by the caller. The engine never touches the pool, never resolves a project path, never spawns a worker. This is the property Phase 119 depends on.
</objective>

From RESEARCH §1 — `scan_versions` columns at migration head 16:

```sql
scan_versions (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,  -- monotonic across all repos
  repo_id       INTEGER NOT NULL REFERENCES repos(id),
  started_at    TEXT NOT NULL,                       -- ISO 8601
  completed_at  TEXT,                                -- nullable (NULL = in-flight)
  quality_score REAL                                 -- nullable; added by mig 015
)
```

From RESEARCH §2 — diff-relevant columns:

```
services columns to PROJECT for diff:
  KEY:    repo_id, name
  FIELDS: root_path, language, type, owner, auth_mechanism, db_backend, boundary_entry, base_path

connections columns (must JOIN to services for source_name + target_name):
  KEY:    source_name (via JOIN), target_name (via JOIN), protocol, method, path
  FIELDS: source_file, target_file, crossing, confidence, evidence, path_template
```

From RESEARCH §5 — selector resolution precedence (apply in this exact order):

```
1. /^\d+$/                             -> integer ID
2. /^HEAD(?:~(\d+))?$/                 -> HEAD or HEAD~N
3. /^(\d{4}-\d{2}-\d{2})(T.*)?$/       -> ISO 8601 date or full timestamp
4. fall through                        -> branch name (requires projectRoot)
```

From RESEARCH §5.4 — branch resolver MUST use `execFileSync` (NOT shell `exec`), passing branch name as an argv element. This is a hard constraint — shell-exec'ing user-controlled branch strings is a command-injection vulnerability.

```javascript
import { execFileSync } from 'node:child_process';
const sha = execFileSync(
  'git', ['-C', projectRoot, 'rev-parse', branch],
  { encoding: 'utf8', stdio: ['ignore', 'pipe', 'pipe'] }
).trim();
```

From RESEARCH §1.2 — ISO timestamps sort lexicographically same as chronologically, so `WHERE completed_at <= ?` works directly on the TEXT column without `datetime()` casting.

From RESEARCH §4.2 — diff key composition: use `JSON.stringify([repoId, name])` for service keys and `JSON.stringify([srcName, tgtName, protocol, method ?? '', path ?? ''])` for connection keys. NULL method/path default to empty string for stable Map keys.

<background>
Phase 115 is split into two plans because Phase 119 (`/arcanon:diff --shadow`) imports the diff engine without the CLI. Keeping the engine in its own module (this plan) makes that import a one-line change for Phase 119. Plan 115-02 ships the CLI handler + slash command + bats E2E that depend on this engine.

Per RESEARCH §3, no existing helper lists `scan_versions` or resolves a selector — both are net-new in this plan. The only existing model is `getScanQualityBreakdown(id)` at `query-engine.js:1444-1474` (single-row lookup of one scan). The new module operates one level above QueryEngine: it takes a raw `Database` handle and queries directly, so it doesn't need to extend QueryEngine.

Per RESEARCH §8 — the cross-phase contract for Phase 119 is the engine signature: `diffScanVersions(dbA, dbB, scanIdA, scanIdB)`. Tasks below explicitly forbid patterns that would break this (no global state, no pool lookup, no projectRoot parameter on the engine itself).

Per RESEARCH §1.1 — `scan_versions.id` AUTOINCREMENT is the primary diff key; multi-repo scans produce multiple rows but the AUTOINCREMENT ordering preserves global "newest scan" semantics for HEAD~N. No per-repo reconciliation needed.
</background>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-115-01-01 | I | branch resolver | mitigate | `execFileSync` with argv array; never `exec`/`execSync`/shell. Test 14 in Task 1 enforces via grep regression |
| T-115-01-02 | I | SQL parameters | mitigate | All queries use parameterised `?` placeholders (`db.prepare(...).get(value)`) — no string concatenation of user input into SQL |
| T-115-01-03 | T | scanIdA / scanIdB type confusion | mitigate | Resolver returns `{scanId: number}` (Number-coerced); engine receives integers. SQL `?` binding rejects non-integers in better-sqlite3 |

| T-115-01-06 | I | engine accidentally writes to DB | mitigate | Test 15 in Task 2 snapshots row counts pre/post. Module source contains zero `INSERT`/`UPDATE`/`DELETE` strings |
